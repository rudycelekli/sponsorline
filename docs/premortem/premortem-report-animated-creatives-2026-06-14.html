Premortem · Gary Klein method

Sponsorline Animated Creative Format

Frame: It is December 2026. The animated sponsor-line creative format has failed. Working backward to explain why.

8 failure modes investigated · 1 parallel agent per mode · synthesized 2026-06-14

Synthesis

Most likely failure

The animation doesn't actually animate, and renders as garbage on a meaningful slice of terminals.

Two architectural facts make this near-certain: (1) frame advancement is hostage to a redraw cadence the host owns and Sponsorline never measures, and (2) color is gated on !NO_COLOR alone, with truecolor emitted unconditionally and no COLORTERM/TERM/isTTY probe. On most real machines the status-line "animation" is a frozen or jittering single frame; on non-truecolor or non-TTY hosts it leaks raw escape codes. Fix first: most probable, and it silently nullifies the premise (motion you paid a premium for that does not move).

Most dangerous failure

A photosensitive-epilepsy strobe served through valid, allowlisted inventory.

Low probability, catastrophic impact. The validator caps fps, frame count, and loopMs, none of which bound flash frequency or luminance delta. A two-frame 30fps black/white frames creative, or a 1ms-loop high-chroma pulse, is a WCAG 2.3.1 violation that passes validation today. Because rendering is a pure local transform, there is zero server-side telemetry that it happened until someone is harmed. For a network whose entire brand is trust and safety, one viral incident plus a demand letter is existential.

Hidden assumption

That Sponsorline controls the rendering environment.

It controls none of the three things this format depends on: the host owns redraw cadence (so motion may not happen, or may busy-loop visibly), the terminal owns color/escape capability (so output may corrupt), and the developer owns perception (so the same motion reads as either annoyance or surveillance). Every one of the 8 failure modes traces back to assuming control over an environment that belongs to someone else.

The Revised Plan

  1. Ship motion OFF by default in the status line. Static is the contract opted-in developers actually agreed to. Make motion explicit opt-in via a documented flag surfaced in init output. Defuses annoyance churn, the "busy-loop = tracking" perception, and removes the default redraw loop entirely. → maps to #3, #7
  2. Detect terminal capability before emitting ANSI. Gate on a real probe: COLORTERM=truecolor for 24-bit, downgrade to 256-color, then to none; respect isTTY so non-interactive sinks never get escapes. Never emit truecolor unconditionally. → maps to #1
  3. Stop selling/charging "premium" until lift is measured. Remove the "premium inventory class" framing from campaign.ts; animated and plain clear the same auction. Keep the size/effect caps, justified as safety regardless of pricing. Introduce a tier only after a controlled A/B shows real lift. → maps to #4, #8
  4. Add WCAG 2.3.1 flash-safety validation. Reject pulse with effective period <333ms (a loopMs floor), and reject frames whose consecutive relative-luminance delta exceeds ~10% more than 3 times/sec. Cap chroma/contrast swing. Add tests that the 30fps black/white strobe and the 1ms-loop red pulse are REJECTED. → maps to #6
  5. Feature-flag or shelve frames/watch until an advertiser actually wants frames. Do not carry the most security-sensitive, newline-preserving half of the schema in the audited default surface for zero traffic. Keep effect as the live format; gate frames behind a flag that is off by default. → maps to #5, #8

Pre-Launch Checklist

  • Terminal capability matrix test across tmux (no Tc), xterm, Git Bash, VS Code integrated, JetBrains, SSH, and a CI log-pipe: confirm zero raw-escape leakage and zero corrupted renders. (#1)
  • Instrument inter-redraw interval and frame-identity rate in a canary before charging anything: only call motion "real" if frame-identity is <30% and median redraw interval beats the frame period. (#2, #4)
  • Flash-safety validator with red-team tests: the black/white 30fps strobe and the 1ms-loopMs high-chroma pulse must fail validation; commit these as test cases. (#6)
  • Default-off motion + documented opt-in: verify init output explains the motion flag and SPONSORLINE_REDUCED_MOTION, and that a fresh install shows a static line. (#3, #7)
  • Explicit keep/shelve decision on frames/watch: if kept, ship behind an off-by-default flag, excluded from the default audit surface until there is advertiser demand and proven effect-lift. (#5, #8)

Failure Modes · Deep Dives

FAILURE 01

Terminal incompatibility and corruption

Likelihood Impact

Decoder gates ANSI on !NO_COLOR alone, then emits truecolor unconditionally with no TERM/COLORTERM/isTTY probe. Non-truecolor hosts (old tmux, basic SSH, Git Bash, JetBrains) leak literal escape text; shimmer emits one escape per char, becoming dozens of fragments. Non-interactive capture accumulates megabytes of escapes. A "Sponsorline broke my terminal" screenshot spikes opt-out before the zero-egress story is heard.

Underlying assumption

That honoring NO_COLOR is equivalent to detecting terminal capability.

Early warning signs
  • Opt-out among non-truecolor/non-TTY environments >2x the truecolor-TTY baseline in 30 days.
  • Host redraw interval exceeds the creative's frame period (e.g. >150ms against 83ms).
FAILURE 02

Host doesn't redraw often enough to animate

Likelihood Impact

The demo's motion came from watch's own 12fps loop, not the status line being sold. In production the status line renders once per host invocation; the host redraws on events, not a cadence. Engineers saw a frozen frame for 20-40s then a jump. Advertisers diffed rendered bytes and found 60-80% of impressions byte-identical to the prior frame. The premium tier got reclassified as static; refunds issued.

Underlying assumption

That an event-driven host redraws often and regularly enough to look like animation, when frame advancement is hostage to a cadence Sponsorline neither controls nor measures.

Early warning signs
  • Median gap between runStatusline calls >~200ms (stutter) or tens of seconds (static).
  • >~30% of consecutive emissions have identical decoded bytes.
FAILURE 03

Motion perceived as annoying / distracting

Likelihood Impact

Shimmer keyed to input.now runs on every host repaint, and Claude Code repaints on nearly every keystroke. The sweep never stops, pulsing in peripheral vision through every debugging session. The static-line cohort had bargained for a quiet, dead line in exchange for getting paid; motion broke that bargain unilaterally. Supply eroded fastest among the most engaged power users, exactly the high-value inventory.

Underlying assumption

That developers who tolerated a static line had consented to attention capture, when they had consented precisely to its absence of motion.

Early warning signs
  • SPONSORLINE_REDUCED_MOTION adoption climbing among retained users.
  • Uninstall rate for the animated cohort exceeding static baseline within 7 days of exposure.
FAILURE 04

"Premium animated inventory" is a billing fiction

Likelihood Impact

validateAnimated calls animation a "premium inventory class," but campaignToBidder flattens every campaign into one bidCents feeding the same Vickrey auction. "Premium" lived only in a validation message. Advertisers asked for the lift report; there was none, because billing seals the creative as an opaque string. Buyers reverted to plain-text bids that cleared the identical auction at the identical price.

Underlying assumption

That labeling a creative "premium" translates into willingness-to-pay, without any billing tier or lift measurement to make it real or provable.

Early warning signs
  • Effective CPM for animated vs plain within the same cohort stays within noise (<5% delta).
  • Advertiser renewal rate on animated campaigns falls below plain-text.
FAILURE 05

sponsorline watch is a feature nobody uses

Likelihood Impact

September metrics: watch invocations were 0.3% of installs, 80% single-run. Developers use Claude Code to write code, not watch ads. Meanwhile the frames half (cols/rows/fps caps, 240-frame limit, per-frame sanitization) was the most security-sensitive code in the project, the only path emitting multi-row output with newlines preserved. Every audit re-litigated terminal-injection risk on a feature with no audience.

Underlying assumption

That building a surface for frames would create demand for frames, when the developer had no reason to look at it.

Early warning signs
  • watch invocations per install per week under ~1, repeat-run under 10%.
  • Zero frames-kind creatives submitted; effect is 100% of served inventory.
FAILURE 06

Advertiser abuse via motion (accessibility)

Likelihood Impact

The validator did what it was told (fps<=30, frames<=240, loopMs>0). An advertiser shipped a valid two-frame fps:30 creative alternating pure black and white: a 15 Hz strobe, 5x the WCAG 2.3.1 ceiling. pulse with loopMs:1 became a high-frequency red flash. Pure local transform meant zero server-side telemetry. A photosensitive developer had a seizure; "Sponsorline gave my coworker a seizure" hit front page in 48 hours alongside a demand letter.

Underlying assumption

That enforcing structural caps (fps, frame count, loopMs) equals enforcing perceptual safety, when flash hazard is a function of luminance-delta and frequency, neither measured.

Early warning signs
  • Static analysis finds any creative with luminance transitions/sec >3 (or pulse period <333ms): count must be 0.
  • Share of approved creatives with frame-to-frame luminance delta >10% AND flash period <333ms rising.
FAILURE 07

Trust erosion through perception

Likelihood Impact

To make pulse/shimmer move, the status line redraws ~every 80ms. A skeptic running strace/dtruss saw the process waking dozens of times a second. The decode was pure and local, but "constantly redrawing" looks identical to a beacon to someone primed to distrust. The source-available proof became a liability: skeptics read the code, found a render loop, posted "told you so." Churn concentrated among the privacy-skeptical supply the network couldn't replace.

Underlying assumption

That trust comes from the code being provably pure, when it actually comes from the tool behaving inert, and a wall-clock-driven redraw loop visibly is not inert.

Early warning signs
  • Rising SPONSORLINE_REDUCED_MOTION=1 adoption plus uninstall spike within 48h of first animated render.
  • Any external observation of redraw frequency/CPU wakeups framed as suspicion.
FAILURE 08

Complexity outran payoff

Likelihood Impact

~885 lines of source-plus-test for animation, all clean and well-tested. None of it was wrong; it was never needed. Lift over a static line was never proven, because the format predated any traffic to A/B against. At the first enterprise security review a reviewer opened creative.ts, saw raw truecolor generation and a frames kind emitting multi-row grids with newlines kept, and escalated. The dual control-char regexes became a three-meeting fixation. The format became a tax: every audit reopened the ANSI/decoder surface.

Underlying assumption

That animation would deliver enough engagement lift to justify a permanent expansion of the security-audit surface, believed strongly enough to build on but never measured.

Early warning signs
  • Frames-kind near 0% while effect-kind ~100%: half the audited surface carries no traffic.
  • Enterprise reviews citing creative.ts/ANSI average 2+ extra cycles vs plain-text.
